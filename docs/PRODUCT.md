# MOZI PRODUCT.md — Design Context

> Companion to `docs/DESIGN.md` (Impeccable-style project context). Answers "who is
> this for and how should it feel" so design decisions are grounded, not generic.

## Audience
One operator getting real work done on a real machine — a technical professional
(developer, analyst, PM, researcher) using MOZI as a personal Agent OS. Not a mass
consumer, not an enterprise buyer browsing a marketing site. They value density,
speed, and truthfulness over hand-holding or delight-for-its-own-sake.

## Register & Voice
- **Calm, precise, professional.** Confident without hype. No exclamation-mark energy,
  no marketing adjectives, no emoji.
- **Truthful over reassuring.** Surface real runtime state; never fake success, progress,
  or capability. (Mirrors the repo constitution.)
- **Dense but legible.** Assume an expert reader; don't pad. But never sacrifice clarity —
  product proof (chat, tasks, artifacts, code, tables) must always read cleanly.

## Tone in the UI
Quiet, warm-dark, utilitarian. The interface is a workbench, not a showroom. Styling
serves comprehension; it never competes with the operator's content.

## Anti-references (what MOZI is NOT)
- Not a neon/gradient "AI startup" landing aesthetic.
- Not a playful consumer chat app (no mascots, emoji reactions, confetti).
- Not an enterprise dashboard drowning in cards, chrome, and shadows.

## Applies to generated deliverables
When MOZI produces documents/decks/pages for the user, they inherit this register:
professional, restrained, legible, no AI-slop tells. A deck MOZI generates should look
like a competent human made it, not like a template generator filled in blanks.
