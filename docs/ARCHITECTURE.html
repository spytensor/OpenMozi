Autonomous Agent System Architecture

墨子 MOZI

自主进化的智能体操作系统
Architecture v1.0 · February 2026 · Designed by Chaojie

设计哲学

墨子系统以"兼爱非攻"的实用主义为内核：每一层只做自己该做的事，不侵入其他层的职责。系统追求的不是功能的堆叠，而是模块的纯粹——每个组件都可以被独立替换、升级、甚至重写，而不影响整体运行。

核心理念：Brain只发意图，不发命令。SubAgent是可处置的临时工，干完即走，表现好才留下。一切操作必须有超时，一切状态必须可追溯。

Brain ≠ Worker 主Agent永远不被阻塞，它是调度者，不是执行者
Intent, Not Command Brain发出语义意图，TEL翻译成精确指令
Disposable Agents SubAgent挂了就杀掉重建，不尝试恢复corrupted进程
Everything Has a Timeout 任何操作、任何层级，没有例外

Layer by Layer
逐层设计说明
L0
USER INTERFACE
用户接入层

纯粹的消息收发层，不做任何业务逻辑。每个channel（Telegram、WhatsApp、API、CLI）都是一个独立的adapter，职责只有两件事：把用户消息标准化为统一格式送入L1，把L1的输出渲染成channel-specific的格式回复用户。

Channel adapter是无状态的——挂了重启即可，不丢任何上下文。新增一个channel只需要写一个adapter，不触碰系统其他部分。

标准消息格式
{ source: "telegram", user_id: "xxx", timestamp: ISO8601,
  type: "text|file|voice", content: "...",
  reply_to: null | message_id, channel_meta: {...} }
L1
TRIGGER & SESSION GATEWAY
触发与会话网关

系统的"前台"——所有事件（用户消息、定时任务、webhook、subagent状态变更）都从这里进入。它的核心职责是排序、去重、聚合，确保Brain收到的永远是干净的、有优先级的事件流。

会话状态机

三个状态：IDLE（等待输入）→ WORKING（已派活，用户新输入进pending queue）→ RESPONDING（组织回复，先处理pending用户消息，再批量汇报subagent状态）。主Agent永远不会因为在"干活"而无法响应用户。

通知聚合

SubAgent的状态变更不是即时推送给用户的。设300ms debounce窗口，同一窗口内的多个状态变化合并为一条结构化通知。避免OpenClaw那种"消息乱飞"的问题。

优先级

用户新输入 > 任务失败 > 任务完成 > 进度更新 > 系统通知。Brain总是先响应人。

L2
BRAIN (Mozi Core)
大脑层 · 墨子核心

系统的灵魂。使用Opus级别模型，职责严格限定为：理解意图、分解任务、派发工作、验收结果。Brain绝不自己执行任何tool call，绝不自己写代码，绝不自己操作浏览器。它是"团队领导"，只做决策。

Intent Planner

接收用户请求，判断是简单对话（直接回复）还是需要执行的任务（分解为task DAG）。对话类请求不进入task系统，零延迟回复。

Task Decomposer

复杂任务拆解为DAG（有向无环图），定义task间的依赖和阻塞关系。每个task节点包含：目标描述、验收标准、约束条件（token budget、timeout、权限范围）。不告诉subagent"怎么做"，只告诉"做什么"和"做到什么程度"。

SubAgent Lifecycle Manager

管理subagent的完整生命周期。两类agent共存：

Preset Agent（常驻专家） 预定义的specialist，有固定的system prompt和tool权限。如code-reviewer、test-runner。启动成本低，复用度高。
Dynamic Agent（临时外包） 主Agent按需构建的临时agent。Brain定义其角色prompt和权限，spawn为独立进程，任务完成后释放。如果同一类Dynamic Agent被反复spawn且表现良好（spawn_count > N && success_rate > threshold），自动promote为Preset Agent。
Quality Gate

接收SubAgent返回的Result Envelope，对照task的验收标准判断：accept（关闭task）、reject+feedback（要求重做）、escalate（上报用户决策）。这是整个系统质量的最后一道关。

L2↔
BRAIN ↔ SUBAGENT PROTOCOL
主从通信协议

Brain和SubAgent之间通过消息队列（Redis Streams / ZeroMQ）通信，不是直连。SubAgent crash不影响Brain。通信内容严格约束——Brain不看SubAgent的工作过程。

下派：Task Brief
{ task_id, objective: "做什么",
  done_criteria: "验收标准",
  constraints: { token_budget, timeout, allowed_tools, forbidden_paths },
  context_refs: ["file://path", "memory://key"] }

// 不包含"怎么做"。SubAgent自行决定方法。
执行中：仅两种上报

BLOCKED（需要权限/澄清/依赖未满足）或 MILESTONE（关键节点完成）。其他时候闭嘴干活。

完成：Result Envelope
{ task_id, status: "success|failed|partial",
  output: ["file://diff", "artifact://url"],
  summary: "一句话总结",
  cost: { tokens, tool_calls, elapsed_time },
  issues: [] }

// 固定大小，不随内部工作复杂度膨胀。
L3
TOOL EXECUTION LAYER (TEL)
工具执行层

系统的"双手"。接收来自SubAgent的语义意图（intent），翻译成精确的工具命令并执行。这一层的存在意义：即使上层模型的tool calling能力弱，只要TEL够强，系统就稳定运行。

TEL实现了双向解耦——上层换模型不影响工具调用，下层加新工具不需要改Agent的prompt。

处理流水线

Intent Router（选工具）→ Param Validator（Pydantic schema校验）→ Sandbox Manager（决定执行环境）→ Executor（实际执行）→ Result Formatter（统一输出格式）→ Error Handler（失败处理）

Tool SLA 注册

每个工具注册时必须声明：硬超时、软超时、重试策略、降级方案、执行沙箱类型。TEL根据SLA做自动化的超时管理和故障恢复。

自纠错链路

Schema校验失败 → 自动修正参数重试 → 执行失败 → 降级到备选工具 → 仍失败 → 压缩错误上下文（不是raw log）上报SubAgent → SubAgent重新规划 → 超过step budget → 上报Brain

L4
CAPABILITY LAYER
能力层

实际的工具和运行时环境。每个capability是一个独立的adapter，对TEL暴露统一接口。新增能力只需写adapter，不触碰上层。

Shell 受限shell执行，可配置网络/文件系统权限
Playwright 双模式：Snapshot（快/轻量）+ Vision（慢/强理解），自动降级
FileSystem 文件读写，支持git-tracked workspace
API Client HTTP/REST/GraphQL，带自动重试和rate limit
Docker 容器级隔离执行环境，SubAgent沙箱
Database SQL/NoSQL适配器，read-only mode可选
Cross-Cutting Modules
横切模块（接口解耦，可独立替换）

MEMORY INTERFACE

记忆接口

对外暴露统一API：store() / recall() / forget() / summarize()。底层实现可以随时替换，Agent层无感知。

三层记忆：短期（当前task context，在进程内存中）、中期（活跃task的planning状态和中间决策，SQLite）、长期（patterns/经验/知识，Markdown + Git）。

预留Vector DB和Graph DB的adapter接口，为未来记忆技术突破做准备。这是最可能被整体替换的模块。

SKILL REGISTRY

技能注册表

Skill = SKILL.md（能力描述 + 输入输出schema + 前置条件）+ 可选脚本。Markdown-first，人可读可审计。

自进化闭环：遇到新任务 → 发现无现成skill → SubAgent写一个 → 测试通过 → 注册到本地registry → 下次自动匹配调用。

运行时按需注入：只把当前task相关的skill schema注入context，其余不加载，避免prompt膨胀。

TOKEN BUDGET MANAGER

令牌预算管理器

实时追踪每一层的token占用。为Brain、每个SubAgent分别维护独立的budget。在spawn SubAgent前检查预算是否充足。

动态触发compaction策略（详见Token Strategy部分）。同时输出cost数据给Observer，供SubAgent评分和成本控制使用。

OBSERVER

观测器

每个task分配TraceID，所有SubAgent操作、tool call、LLM调用都关联到trace。记录token消耗、调用延迟、成功/失败率。

数据格式兼容OpenTelemetry，存储到SQLite（单机）或ClickHouse（规模化后）。这是SubAgent"进化评分"的数据来源。

WATCHDOG

看门狗（独立进程）

完全独立于主系统的守护进程。每5秒检查Brain心跳和所有SubAgent心跳。发现无响应 → graceful restart → 失败则force kill + 从最近checkpoint恢复 + 通知用户。

同时负责全局级别的超时兜底：如果某个task超过最大允许时间仍未完成，直接终止并上报。这是防止整个系统"卡死"的最后一道防线。

SubAgent Token 策略

每个SubAgent启动时只注入：角色system prompt + task brief + 相关文件引用（摘要或路径，不是全文）。启动开销控制在 5-10% window，剩余90%是工作空间。SubAgent不继承主Agent的对话历史。

Multi-Model Token 经济学

主Agent（Opus级）：调用少但精准，承担planning和验收。复杂SubAgent（Sonnet级）：干活主力。简单SubAgent（Haiku/Flash级）：摘要、格式转换、简单生成。TEL层的intent解析：优先用规则引擎，能不用LLM就不用。

关于"墨子"这个名字

墨子（Mozi）是中国战国时期的哲学家和工程师。墨家是古代中国最接近"技术团队"的思想流派——严密的组织结构、实用主义的行动准则、以及"兼爱"的协作哲学。

钜子
墨家领袖，对应 Brain / Mozi Core — 做决策、不干活
墨者
墨家弟子，各有专长，对应 SubAgents — 领命执行、汇报结果
墨守
墨家守城术，对应 Watchdog + 安全边界 — 系统防御的最后一道关
非攻
不做多余的事，对应 设计哲学 — 每层只做自己的事，不侵入其他层
兼爱
模块间平等协作，对应 接口解耦 — 任何模块可被替换而不影响全局
尚同
统一标准，对应 统一的消息格式 + Result Envelope — 通信协议一致
